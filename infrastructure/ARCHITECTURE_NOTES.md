# Architecture Notes — Free Tier Deployment Decisions

> **For portfolio reviewers:** The deployment architecture in this repository differs from what you would use in a real production system. The changes were made exclusively to stay within the AWS Free Tier and avoid any charges. This document explains what was changed, why, and what the production version would look like.

---

## Background

FinFlow is a personal finance planner built as a portfolio project. The full application uses an event-driven architecture on AWS (Lambda, SQS, SNS, RDS, EC2, CloudFront) that maps to real-world fintech patterns. However, deploying this architecture correctly — with proper network isolation — is not free.

During the Terraform phase, a cost analysis revealed that the "correct" architecture would cost **$22–$44/month from day one** due to VPC Interface Endpoints, even within the 12-month AWS Free Tier window. Since this is a personal learning project, the decision was made to adapt the architecture to achieve genuine $0/month during the Free Tier period.

---

## What the original design looked like

The original Terraform design (preserved in [`modules/database/`](terraform/modules/database/) for reference) used:

- **Amazon RDS db.t3.micro** — managed PostgreSQL in a private subnet, no public IP address
- **Lambda functions inside a VPC** — required because Lambda needs to be in the same VPC to reach a private RDS endpoint
- **VPC Interface Endpoints** for SQS, SNS, and SSM — required because a Lambda inside a private VPC has no internet access and cannot call AWS services without either a NAT Gateway or dedicated VPC Endpoints

**Why this is the correct production pattern:**
- RDS in a private subnet is never reachable from the public internet — the attack surface is minimal
- Lambda-to-database connectivity is secured at the network layer, not just by credentials
- Managed RDS handles automated backups, point-in-time recovery, minor version patching, and failover
- SSM Parameter Store is the standard way to inject secrets into Lambda at runtime without hardcoding them in environment variables

**The cost problem:**

| Resource | Why needed | Monthly cost |
|---|---|---|
| NAT Gateway | Lambda in VPC → internet | ~$32 |
| VPC Interface Endpoint (SQS) | Lambda in VPC → SQS | ~$7.30/AZ |
| VPC Interface Endpoint (SNS) | Lambda in VPC → SNS | ~$7.30/AZ |
| VPC Interface Endpoint (SSM) | Lambda in VPC → SSM | ~$7.30/AZ |

With the original 2-AZ configuration: **$43.80/month** just in networking, starting from day one regardless of usage. None of these are covered by the AWS Free Tier.

## What was changed and why

### 1. RDS replaced by PostgreSQL in Docker on EC2

The same EC2 t2.micro instance that runs Celery workers now also runs PostgreSQL and Redis via Docker Compose ([`modules/workers_ec2/user_data.sh`](terraform/modules/workers_ec2/user_data.sh)).

The EC2 Free Tier includes **750 hours/month of t2.micro usage** and **750 hours/month of public IPv4 address usage**. Running one instance 24/7 (730h/month) fits within both limits, making the instance and its Elastic IP completely free for 12 months.

Data is stored on the EC2's EBS root volume (30GB, also Free Tier). The volume persists on instance stop, so data survives reboots. It would be lost only on `terraform destroy` (which is protected by a `lifecycle` rule).

### 2. Lambda removed from the VPC

Without a private RDS endpoint to reach, Lambda no longer needs VPC membership. Lambda functions connect to PostgreSQL and Redis on the EC2's Elastic IP directly over the public internet, authenticated by a 32-character random password generated by Terraform.

Lambda outside a VPC has standard internet access by default, so it can call SQS, SNS, and SSM without any additional networking infrastructure.

### 3. VPC Interface Endpoints removed (SQS, SNS, SSM)

With Lambda outside the VPC, Interface Endpoints are no longer needed. Lambda calls SQS and SNS directly via the standard AWS SDK, which routes over AWS's backbone network. This is how Lambda works by default when not configured with a VPC.

The S3 Gateway Endpoint is kept — it is free and improves performance for EC2-to-S3 transfers (audit log uploads).

### 4. Kafka removed from the EC2 production setup

The EC2 t2.micro has 1GB of RAM. Kafka and its ZooKeeper dependency together consume approximately 512MB of JVM heap, leaving insufficient headroom for PostgreSQL and Redis running concurrently.

Kafka continues to work in the **local development environment** (`docker-compose.workers.yml`). In the cloud deployment, `KAFKA_BOOTSTRAP_SERVERS` is set to an empty string in all Lambda environment variables. The backend already handles this gracefully: the Kafka producer uses `asyncio.create_task()` with a `never_raises` pattern, so a missing Kafka connection is silently ignored without affecting the HTTP response.

## Cost comparison

| Configuration | Monthly cost (Free Tier period) |
|---|---|
| Original design — RDS + Lambda in VPC + 3 Interface Endpoints (2 AZ) | ~$43.80 |
| Original design — 1 AZ only | ~$21.90 |
| **This deployment** | **$0.00** |

After the 12-month Free Tier ends, costs would be approximately:

| Resource | Monthly cost (year 2+) |
|---|---|
| EC2 t2.micro | ~$8 |
| S3 + CloudFront | <$1 |
| Lambda, SQS, SNS | ~$0 (well within always-free tier) |
| **Total** | **~$9/month** |

## What a real production system would look like

| Concern | This portfolio deployment | Production |
|---|---|---|
| **Database** | PostgreSQL in Docker on a shared EC2 instance | Amazon RDS Multi-AZ with automated backups and point-in-time recovery |
| **Network security** | PostgreSQL and Redis ports open to the internet, protected by password only | Database in a private subnet with no internet exposure; access restricted by VPC security groups |
| **Lambda connectivity** | Lambda outside VPC, connects to DB over the public internet | Lambda inside VPC via private subnet; all traffic stays within the AWS network |
| **Secrets at runtime** | Secrets embedded as Lambda environment variables at deploy time | AWS Secrets Manager with automatic rotation; secrets fetched at Lambda cold start |
| **Audit log** | Kafka disabled in production (RAM constraint) | Kafka on a dedicated instance or Amazon MSK; audit trail always active |
| **High availability** | Single EC2 — if the instance goes down, the database and workers go down | Separate managed services (RDS, ElastiCache, MSK), each with redundancy and automatic failover |
| **Backups** | Manual EBS snapshots or none | RDS automated daily backups with configurable retention, point-in-time restore |
