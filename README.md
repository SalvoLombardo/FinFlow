# FinFlow

Personal financial planner with AI — event-driven architecture on AWS.

---

## Why this architecture looks non-standard

> **Read this first.** The deployment described below intentionally deviates from production
> best practices. Understanding the reason prevents misinterpreting it as an error.

The standard way to deploy a Lambda + PostgreSQL system on AWS is:

- Lambda **inside a private VPC** → can only reach the database through internal AWS networking
- PostgreSQL on **Amazon RDS** → managed, in a private subnet, never exposed to the internet
- **VPC Interface Endpoints** for SQS, SNS, SSM → allow Lambda inside the VPC to call AWS
  services without leaving the network (no NAT Gateway needed)

This is the right architecture for production: the database is unreachable from the internet,
all traffic stays within AWS's backbone, and secrets are fetched at runtime from SSM.

**The problem: none of the VPC networking components are covered by the AWS Free Tier.**

| Component | Why it is needed | Monthly cost |
|---|---|---|
| NAT Gateway | Lambda in VPC → internet (alternative to endpoints) | ~$32 |
| VPC Interface Endpoint (SQS) | Lambda in VPC → SQS | ~$7.30 per AZ |
| VPC Interface Endpoint (SNS) | Lambda in VPC → SNS | ~$7.30 per AZ |
| VPC Interface Endpoint (SSM) | Lambda in VPC → SSM | ~$7.30 per AZ |

With two Availability Zones, the correct architecture costs **~$44/month from day one**,
before any application traffic. This is a learning / portfolio project, so the decision was
made to reach genuine $0/month instead.

**The solution applied:**

1. Lambda runs **outside the VPC** — no VPC Endpoints or NAT Gateway needed.
2. PostgreSQL and Redis run in **Docker on the EC2 t2.micro** (the same instance used for
   Celery workers), rather than on a managed RDS instance. EC2 t2.micro is Free Tier eligible.
3. Lambda connects to PostgreSQL and Redis via the **EC2 Elastic IP** over the public internet,
   protected by a 32-character random password generated by Terraform.
4. Kafka is **disabled in the cloud deployment** — the EC2 t2.micro has 1GB of RAM, and Kafka
   + ZooKeeper consume ~512MB of JVM heap, leaving insufficient headroom alongside PostgreSQL
   and Redis. Kafka works fully in the local development environment.

The original production-grade RDS Terraform module is preserved in
[`infrastructure/terraform/modules/database/`](infrastructure/terraform/modules/database/)
for reference. The full trade-off analysis is in
[`infrastructure/ARCHITECTURE_NOTES.md`](infrastructure/ARCHITECTURE_NOTES.md).

## Architecture

```
WRITE PATH                          SCHEDULED PATH
FastAPI Lambda (outside VPC)        Celery Beat
      │                                  │
      ▼                                  ▼
SNS Topic (finflow-events)         Celery Workers
      │
      ├──► SQS projections-queue  ──► Lambda projection-consumer
      ├──► SQS ai-analysis-queue  ──► Lambda ai-consumer
      └──► SQS notifications-queue ──► Lambda notification-consumer

EC2 t2.micro — Elastic IP 52.16.138.76
  └── docker-compose.infra.yml
        ├── PostgreSQL 16   (port 5432)
        └── Redis 7         (port 6379)

AUDIT LOG (local development only)
Kafka Topic finflow.audit ──► consumer ──► audit_archive/ (local .jsonl)
```

All four Lambda functions connect to PostgreSQL and Redis over the EC2 Elastic IP.
SQS, SNS, and SSM are reached directly via the default Lambda internet access —
no VPC Endpoints involved.

## Local development

### Prerequisites

- Docker + Docker Compose
- Python 3.12+
- Node.js 20+

```bash
cp .env.example .env
# fill in SECRET_KEY and ENCRYPTION_KEY
```

### Generate secrets

```bash
# SECRET_KEY
openssl rand -hex 32

# ENCRYPTION_KEY (Fernet — used to encrypt AI API keys at rest)
python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
```

### Start backend + database

```bash
docker compose up
```

Backend at `http://localhost:8000`. Swagger UI at `http://localhost:8000/docs`.

### Run migrations

```bash
cd backend && alembic upgrade head
```

### Start workers (optional — needed for scheduled tasks and Kafka audit log)

```bash
docker compose -f docker-compose.workers.yml up
```

### Start frontend

```bash
cd frontend && npm install && npm run dev
```

Frontend at `http://localhost:5173`. API calls are proxied to `http://localhost:8000` by Vite.

### Run tests

```bash
cd backend && pytest --cov=app tests/
```

## Data model — non-obvious invariants

### Balance chain

Every financial week has an `opening_balance` and a `closing_balance`.
`closing_balance` is **always computed** as `opening_balance + income - expense`; it is
never set directly by the client (the field is read-only in `PUT /weeks/{id}`).

The chain propagates automatically:

```
UserFinancialSettings.initial_balance   ← single source of truth for the starting balance
        │
        ▼
Week 1  opening = initial_balance      closing = opening + income − expense
        │
        ▼
Week 2  opening = Week 1 closing       closing = opening + income − expense
        │
        ▼
  ...
```

`PUT /settings/financial` is the only way to adjust the starting point of the chain.

### Weeks are auto-materialised, never created manually

`POST /weeks` does not exist. A `financial_week` record is created in the database only
when a transaction is added to it (`POST /transactions` derives the week from
`transaction_date` via an internal `get_or_create_week` helper). `GET /weeks` returns
future weeks as virtual projected objects — no DB record is written until a real
transaction lands in that week.

### Goal progress is always computed server-side

Two semantically distinct goal types are supported:

| `goal_type` | What it tracks | Progress formula |
|---|---|---|
| `savings` | How much has been saved since goal creation | `(current_balance − baseline_balance) / target_amount` |
| `liquidity` | Whether the account will reach a minimum balance by a date | `projected_balance_at_target_date / target_amount` |

`baseline_balance` is a snapshot of the current balance captured automatically when a
`savings` goal is created. `progress_pct` (0–100) is always returned by the API and never
accepted as input — the client never calculates it.

## Event routing

Each SQS queue subscribes to `finflow-events` with a filter policy on the `event_type`
message attribute.

| Queue | Event types |
|---|---|
| `projections-queue` | `budget.updated`, `week.closed` |
| `ai-analysis-queue` | `ai.analysis.requested`, `budget.updated` |
| `notifications-queue` | `goal.progress` |

**Events emitted by the API:**

| Endpoint | Event |
|---|---|
| `POST/PUT/DELETE /transactions` | `budget.updated` |
| `POST /insights` | `ai.analysis.requested` |

When `AWS_SNS_TOPIC_ARN` is empty (local dev), every write logs the event as structured JSON
instead of publishing to SNS.

## AI layer

AI analysis is always **asynchronous** — the client never calls an AI provider directly.

```
write event on SNS
      │
      ▼
ai_consumer Lambda
      │  reads user_ai_settings
      │  decrypts API key (Fernet)
      │  builds prompt from real financial data
      │  calls provider (OpenAI / Anthropic / Gemini / Ollama)
      ▼
INSERT into ai_insights
      │
      ▼
GET /api/v1/insights  ← client polls here
```

**Rate limit:** 10 manual `POST /insights` per user per day (Redis INCR + TTL).
API key encrypted at rest with Fernet (`ENCRYPTION_KEY`); never logged or returned.
